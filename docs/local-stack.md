# Local-inference stack — Qwen3-8B / faster-whisper / Kokoro / Pipecat

Sutando today routes voice / chat through cloud models (Gemini Live for realtime,
Claude or the Anthropic API for the core agent). This plan adds a **fully local**
alternative — every component swappable, every component runnable on-host or on
another LAN machine — so an operator with the right hardware can run sutando
without any external API key.

> **Recon completed — see [`local-stack-scope.md`](local-stack-scope.md) for the answers
> to every open question below.** The scope doc supersedes the speculation in this
> plan where they conflict. Top-line corrections from recon:
>
> - **"Turboquant" is a KV-cache technique, not a weight quant.** The actual weight-quant pick is
>   `Qwen/Qwen3-8B-AWQ` (official, Apache 2.0) for HTTP serving, and `Qwen3-8B-GGUF` Q4_K_M
>   for in-process via `LlamaSharp`. TurboQuant lives on the roadmap as a long-context KV
>   optimisation, not a primary pick.
> - **The .NET ecosystem covers every stage natively** (`LlamaSharp`, `Whisper.net`,
>   `KokoroSharp`, Silero ONNX). The DIY-ONNX track in this doc collapses into "use the NuGet."
> - **Pipecat recommendation inverts** — ship Shape B (C# pipeline port in `Sutando.Pipeline`)
>   first, not the Python sidecar. No .NET Pipecat SDK exists and the protocol is pre-1.0.
> - **Aspire adopted** for `Sutando.AppHost` (`AddDockerfile` for the Python services + OTLP
>   dashboard for free + `docker-compose.yml` fallback generated by Aspire 13).
> - **Implementation order re-sequenced** — in-process adapters before HTTP adapters
>   (laptop-end-to-end first, container demo second).

The local stack mirrors the cloud-stack pipeline:

```
mic ──► VAD ──► STT ──► LLM ──► TTS ──► speaker
       (Silero)  (faster-      (Qwen3-8B   (Kokoro
                  whisper)     turboquant)  82M)
```

orchestrated by **Pipecat**-style pipes — either via the upstream Python
Pipecat library running as a sidecar, or by a thin C# port of the same
pipeline pattern living inside `Sutando.Pipeline`.

## Recommended components

| Stage | Component | Why |
|---|---|---|
| **VAD** | Silero VAD (ONNX, ~1 MB) | The same VAD Pipecat / livekit / sutando-upstream already converge on; tiny, accurate, ONNX-friendly. |
| **STT** | faster-whisper (medium → large-v3) | CTranslate2 backend, ~4× whisper.cpp throughput on GPU, ~equivalent on CPU. INT8 quants run in 8 GB VRAM. Mature HTTP wrapper: `faster-whisper-server`. |
| **LLM** | Qwen3-8B (turboquant variant — needs research) | Strong instruction-following for an 8B; Apache 2.0; multilingual. "Turboquant" likely refers to Alibaba's Turbo Engine quants or a third-party AWQ/GPTQ; **a research agent will pin this down before we commit to a serving stack**. |
| **TTS** | Kokoro 82M | Recent open TTS, ~50 MB ONNX, MIT, multi-voice. Pipecat already ships a Kokoro service. ONNX-shippable for in-process use. |
| **Orchestration** | Pipecat (Python sidecar) OR `Sutando.Pipeline` (C# port) | Both are viable — see "Pipecat integration shapes" below. |

Each stage is behind a Sutando interface so the implementation is swappable:

- `ISpeechToText` — `Task<string> TranscribeAsync(ReadOnlyMemory<byte> pcm, AudioFormat in, CancellationToken ct)`
- `ITextToSpeech` — `IAsyncEnumerable<AudioChunk> SynthesizeAsync(string text, VoiceConfig voice, CancellationToken ct)`
- `IChatCompletion` — `IAsyncEnumerable<ChatChunk> StreamAsync(IReadOnlyList<ChatMessage> history, ChatOptions opts, CancellationToken ct)`
- `IVadDetector` — `IAsyncEnumerable<VadEvent> AnalyzeAsync(IAudioSource source, CancellationToken ct)`

The existing `IRealtimeTransport` (Gemini Live today) becomes one of several backends;
a new `LocalPipelineTransport` composes the four interfaces above into the same
event stream the rest of sutando already consumes.

## Deployment modes

Every component supports all four modes through the same .NET interface — the
adapter handles wire-protocol differences.

### 1. Cloud (today)

- LLM: Anthropic / Gemini Live (already implemented).
- STT/TTS: Gemini Live native audio handles both as one round-trip.
- No new work; this is the baseline.

### 2. Remote services (HTTP/gRPC to LAN machines)

- LLM: an OpenAI-compatible endpoint (`/v1/chat/completions`) exposed by
  vLLM / llama-server / TGI / LM Studio running on a beefier LAN box.
  Sutando's adapter is HTTP-only, so it doesn't care which serving stack.
- STT: `faster-whisper-server` HTTP API (same shape as OpenAI's `/v1/audio/transcriptions`).
- TTS: `kokoro-fastapi` HTTP (Kokoro's recommended deployment).
- VAD: in-process (cheap; no need to remote).

Adapters: `OpenAiCompatibleChatCompletion`, `FasterWhisperHttpStt`, `KokoroHttpTts`.

### 3. In-process ONNX (single host, contained)

- LLM: Qwen3-8B via ONNX Runtime (or via `LlamaSharp` over llama.cpp's GGUF —
  benchmark both; ONNX wins on portability, GGUF wins on RAM/perf headroom).
- STT: `whisper.onnx` or `Whisper.net` (whisper.cpp wrapper).
- TTS: `kokoro.onnx` via `Microsoft.ML.OnnxRuntime`.
- VAD: `silero_vad.onnx` via `Microsoft.ML.OnnxRuntime`.

Single process; no containers; works wherever .NET 10 runs and the model files fit.

### 4. Multi-container with .NET Aspire

When the operator wants to run the local stack as a fleet of containers (per-component
GPU pinning, independent restart, clear ports), an Aspire AppHost orchestrates everything:

```text
Sutando.AppHost (.NET Aspire)
├── llm           — vLLM container, Qwen3-8B AWQ, exposed on :8001
├── stt           — faster-whisper-server container, exposed on :8002
├── tts           — kokoro-fastapi container, exposed on :8003
├── pipecat       — optional pipecat-sidecar container if we go with the Python route
├── sutando-core  — the sutando agent itself (also a container)
└── observability — built-in Aspire OTLP collector + dashboard
```

Aspire gives us service discovery, health checks, a dashboard, and OTLP wiring
for free. Each container can be swapped out (`vllm` ↔ `llama-server`) without
code changes to sutando.

## Pipecat integration — two shapes

Pipecat is Python; we can't just `using PipecatNet`. Two viable shapes:

### Shape A — Pipecat-as-sidecar

We keep Pipecat as a Python service. Sutando connects to it over WebSocket
(Pipecat's native client protocol). Pipecat owns the realtime loop: VAD, STT,
LLM, TTS, audio frames. Sutando provides bot instructions, tool definitions
(via the function-calling adapter), and consumes transcripts + tool calls back.

- **Pros**: zero pipeline code on our side; Pipecat's existing service catalog
  (Cartesia, Deepgram, Riva, ElevenLabs, OpenAI, Anthropic, Gemini, Kokoro,
  faster-whisper, etc.) becomes ours for free.
- **Cons**: Python sidecar to install / manage; less control over the wire;
  duplicate transcoding when sutando also has its own audio loop.

### Shape B — `Sutando.Pipeline` (C# port)

We port the pipeline-pipe pattern to C# — `IPipelineStage`, `Pipeline.Run`,
backpressure, frame contracts — and reuse Pipecat's design at the wire level
(same frame shapes, same STT/LLM/TTS interfaces) without the runtime.

- **Pros**: single-language stack; no Python install; ergonomic for .NET
  consumers; integrates cleanly with `Sutando.Realtime`'s event stream.
- **Cons**: re-implementation work; we lag Pipecat's service catalog; risk of
  semantic drift from upstream.

**Recommendation:** ship Shape A first (lower risk, faster). Ship Shape B
once we have two real consumers asking for it, so the design is grounded.

## Hardware floor

For reference, the recommended local stack at usable quality:

- **Minimum (CPU-only):** Qwen3-1.5B (downsize), faster-whisper `base`, Kokoro
  82M, Silero VAD — runs on a recent laptop, latency 2-4 s per turn.
- **Recommended (GPU):** Qwen3-8B turboquant (~6 GB VRAM), faster-whisper
  `medium` INT8 (~2 GB VRAM), Kokoro 82M (CPU is fine), Silero (CPU). ~14 GB
  total VRAM headroom; runs on a single RTX 4070 or M2 Pro / Max.
- **Generous:** Qwen3-32B (Q4_K_M), faster-whisper `large-v3` INT8, Kokoro
  + voice cloning. 24-48 GB VRAM (RTX 4090 / M3 Max / dual RTX 3090).

## Phases (proposed)

Mapped onto the existing roadmap as new tasks #15-#20:

1. **Recon** — pin Qwen3 quant variant, decide Pipecat-vs-port, decide
   ONNX-vs-GGUF for the in-process LLM path. Writes `docs/local-stack-scope.md`.
2. **`Sutando.LocalInference.Abstractions`** — define `ISpeechToText`,
   `ITextToSpeech`, `IChatCompletion`, `IVadDetector`, audio frame contracts.
3. **HTTP adapters** — `FasterWhisperHttpStt`, `KokoroHttpTts`,
   `OpenAiCompatibleChatCompletion`. These cover deployment modes 2 and 4.
4. **In-process ONNX adapters** — `WhisperOnnxStt`, `KokoroOnnxTts`,
   `SileroVadOnnx`, `QwenOnnxChat` (or `LlamaCppChat` if GGUF wins). These
   cover mode 3.
5. **`Sutando.Pipeline`** (Shape B) — port the Pipecat pipeline pattern to C#
   so we can drive the stack without a Python sidecar. **Or** ship a Pipecat
   sidecar adapter first if the recon recommends Shape A.
6. **`Sutando.AppHost`** — .NET Aspire orchestration project. Container
   definitions for vLLM, faster-whisper-server, kokoro-fastapi, optionally
   Pipecat, plus the sutando agent itself. OTLP wiring for observability.
7. **End-to-end demo** — a `sutando voice --local` mode that boots the full
   pipeline and routes voice calls through it without touching the cloud.

## Open questions for recon

These need answers before phase 2 starts. The recon agent (see follow-up)
will return a `docs/local-stack-scope.md` covering each.

1. **Which Qwen3-8B quant is "turboquant" pointing at?** Alibaba's Turbo
   Engine quants? AWQ-W4-G128? GPTQ? GGUF Q4_K_M? Answer determines serving
   stack (vLLM vs llama-server vs ONNX).
2. **ONNX feasibility for Qwen3-8B.** Does an ONNX export exist? Quality vs
   GGUF Q4_K_M? Inference latency on a reference machine?
3. **faster-whisper-server vs whisper.cpp vs Whisper.net.** Which gives us
   the cleanest cross-platform STT story for both in-process and HTTP modes?
4. **Kokoro deployment.** Pipecat's Kokoro service vs `kokoro-fastapi` vs
   raw ONNX — what's the LOC budget for in-process vs HTTP?
5. **Pipecat .NET interop.** Is there a maintained .NET client? Otherwise:
   what's the simplest WebSocket framing we'd need to implement?
6. **Aspire vs docker-compose.** Aspire's value proposition is service
   discovery + OTLP + dashboard; is that worth the dependency vs a plain
   `docker-compose.yml`?
